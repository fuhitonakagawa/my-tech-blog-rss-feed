viable/strict/1784880052: Skip non-fd shareable_handle in ExpandableSegment cleanup (#190860)
Release notes from pytorch
ExpandableSegment::unmapHandles releases each handle's exported shareable handle with close(std::get(*h.shareable_handle)). But shareable_handle is a std::variant that also holds CUmemFabricHandle for fabric segments, so once a fabric segment has been shared (share() caches the exported CUmemFabricHandle), destroying it makes std::get throw std::bad_variant_access from the destructor, which calls std::terminate.Use std::get_if so only POSIX-fd handles are closed; fabric handles (plain values with nothing to close) are skipped.Test: inspection-level fix. Reproducing the crash needs a fabric-capable multi-process setup and no such CI job exists. std::get_if returns nullptr for a variant holding CUmemFabricHandle, so close() is skipped instead of throwing.#190860https://github.com/ngimel, https://github.com/Skylion007

viable/strict/1784877062: [Inductor] Skip folding inf/nan to int in index propagation (#190427)
Release notes from pytorch
Fixes #190421WhyIndex propagation constant-folds dtype casts with Python's int(), which has no value for inf/nan, so compiling e.g. sum(log(zeros), dtype=torch.int32) crashes with OverflowError at compile time.HowSymPyOps.to_dtype returns NotImplemented for a non-finite constant cast to an integer dtype, following the existing pattern in mod/remainder/minimum; index propagation falls back to the wrapped handler, so the cast is emitted by codegen with the backend's runtime semantics (matching eager) instead of being folded at compile timeAdd a regression test with sum(log(zeros), dtype=torch.int32)Pull Request resolved: #190427https://github.com/eellison
